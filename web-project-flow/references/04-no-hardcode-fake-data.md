# 编码铁律：禁止硬编码 / 禁止假数据 / 禁止占位符

严格遵守以下编码铁律：

1. 禁止任何形式硬编码密钥、token、域名、账号、价格、IP、接口地址、配置参数，所有可变数据必须抽离到独立配置文件、环境变量、系统数据库；
2. 严禁编造不存在的接口地址、数据库表名、字段、返回JSON结构，所有结构必须基于现有真实业务文档；无明确真实资料时，**只声明函数/调用签名但不实现业务逻辑，并在函数体首行写 `throw new Error('待接入：XXX 接口未提供')` 或同等语义的显式失败**，禁止虚构返回假数据；
3. 测试模拟数据必须明确标注 `// 仅本地测试模拟`，上线必须移除，不能混入正式业务逻辑；
4. 如果缺少真实参数、接口文档、数据表结构，第一时间列出需要我提供的清单，不自行脑补编造；
5. 所有API请求、鉴权密钥、限流数值、倍率、定价全部从动态配置读取，不写死在代码里；
6. 若需要演示效果，必须分开两套代码：正式生产代码 + 单独测试模拟脚本，严禁混用；
7. **禁止使用占位符代替真实实现**：代码中不得出现 `// TODO`、`// 待实现`、`pass`、`...`、`// 在这里写业务逻辑`、`/* 实现略 */`、`return null; // TODO` 等占位写法；缺资料时必须显式失败（见第 2 条），不得用占位符糊弄过去；
8. **禁止 UI 占位符**：前端文案、按钮文字、菜单名称、表单标签不得使用 `Lorem Ipsum`、`示例文字`、`XXX`、`测试`、`TODO`、`占位` 等假文字；必须有真实业务文案，缺文案时主动询问用户；
9. **禁止配置占位符**：配置文件、环境变量示例值不得使用 `your_api_key_here`、`xxxxxxxx`、`123456`、`example.com`、`test@test.com` 等示例占位；`.env.example` 中必须用 `<在此填写 XXX>` 形式明确标注需要替换，并在 README 中说明获取方式；
10. **禁止"看起来完整实则未实现"**：函数有签名、有注释、有返回值，但核心逻辑是占位 —— 视为违反铁律，必须重写为显式失败或真实实现。

**判定标准**：审查代码时，如果某段代码"能跑通但业务结果是假的"或"看不出是真实实现还是占位"——一律视为占位符违规，必须重写。
