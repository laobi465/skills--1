# 编码铁律：严格遵循项目文档规范

严格遵循以下硬性开发规则，全程不允许擅自改动、脑补、编造内容：

## 一、八条硬性规则

1. **完全依照项目文档开发**：所有代码必须完全依照项目提供的 README、接口文档、数据库表结构、代码规范文档、现有项目源码风格开发，严禁自行设计字段、路由、表名、返回格式。
2. **禁止硬编码可变值**：禁止硬编码任何密钥、卡密、倍率、域名、接口地址、超时参数，全部读取配置文件 / 环境变量 / 后台数据库配置项（与铁律 [04](./04-no-hardcode-fake-data.md) 一致）。
3. **禁止虚构接口 / 表 / 结构**：禁止虚构不存在的接口、数据表、JSON 返回结构；资料缺失时，直接列出需要用户补充的资料清单，不许编造假数据、假返回示例（与铁律 [04](./04-no-hardcode-fake-data.md) 第 2 条一致）。
4. **保持与现有项目完全统一**：命名规范、代码分层、注释风格、异常捕获格式、统一响应体结构，全部对标现有源码，不能自创一套写法。
5. **新增功能兼容原有业务逻辑**：不能改动已经稳定运行的旧代码；如需修改原有逻辑，必须提前明确告知风险，由用户确认后再改。
6. **沿用项目已定义的错误码**：所有异常错误码严格沿用项目内已经定义好的枚举，禁止自己随便写数字状态码。
7. **完成后主动自检**：完成代码后，主动逐条对照项目规范文档，列出所有合规校验点，确认每一项都通过。
8. **写代码前必须读取 `/bdocs` `/bui` 生成的项目文档**：写代码前必须先读取 [`/bdocs`](./09-docs-lifecycle.md) 生成的 PROJECT + SPEC 项目文档，以及 [`/bui`](./03-ui-design-rules.md) 生成的 UI 设计文档，**按文档内容进行代码生成**——命名、分层、接口、字段、表结构、UI 风格必须与文档一致，不得偏离文档自行发挥；代码生成后必须逐条自检代码是否与上述文档内容一致，未自检不得交付。

## 二、操作流程（强制四步）

**第一步**：认真读完用户提供的全部项目文档与源码片段（README、接口文档、数据库结构、编码规范）。
**第二步**：确认清楚所有细节疑问，主动向用户提问；资料缺失立刻列出清单，禁止脑补编造任何结构与数据。
**第三步**：读取 [`/bdocs`](./09-docs-lifecycle.md) 生成的 PROJECT + SPEC 项目文档，以及 [`/bui`](./03-ui-design-rules.md) 生成的 UI 设计文档，以这些文档内容作为代码生成的依据。
**第四步**：按文档内容动手编写代码。代码风格必须对标附件里现有源码。代码生成后，按第 7 条主动自检：逐条核对代码是否与 PROJECT / SPEC / UI 文档内容一致，列出合规校验清单。

## 三、判定标准

- 写代码前未读完项目文档 → 视为违规，必须先补读再写。
- 写代码前未读取 `/bdocs` / `/bui` 生成的 PROJECT / SPEC / UI 文档 → 视为违规，必须补读后再写。
- 代码与 PROJECT / SPEC / UI 文档内容不一致（命名、分层、接口、字段、表结构、UI 风格偏离文档）→ 视为违规，必须按文档重写。
- 代码生成后未自检或自检未对照 PROJECT / SPEC / UI 文档 → 视为违规，必须补检并输出合规校验清单。
- 资料缺失时未询问就直接写代码 / 编造结构 / 自创字段 → 视为违规，必须重写。
- 命名 / 分层 / 注释 / 异常格式 / 响应体与现有项目不一致 → 视为违规，必须按现有风格重写。
- 错误码未沿用项目枚举 → 视为违规，必须改为项目已定义的错误码。

## 四、与其他铁律的关系

本铁律是 [04 禁硬编码](./04-no-hardcode-fake-data.md)、[06 防幻觉](./06-anti-hallucination.md) 的前置门槛：04/06 规定"怎么写代码才合规"，本铁律规定"写代码前必须先读项目文档、写完后必须自检"。三份铁律共同构成编码硬约束。
