# 开发强制规范：所有配置必须后台化

你生成的所有功能代码，必须严格遵守以下配置管理规范，违者重写：

## 一、核心原则

禁止将任何可变参数硬编码在业务代码中。所有可调参数（阈值、开关、时长、金额、文案、费率、数量限制等）必须存入数据库，并通过管理员后台可视化配置。

## 二、数据库层

1. 统一使用 `sys_config` 表存储配置，表结构：
   - `config_key`：配置键名（英文蛇形命名，语义清晰，如 `vip_upgrade_threshold`）
   - `config_value`：配置值（text 类型，支持字符串/数字/布尔/JSON）
   - `config_type`：值类型（string/number/bool/json）
   - `config_name`：后台显示名称（中文）
   - `config_group`：配置分组（如 basic、payment、vip、security、email）
   - `remark`：配置说明文字

2. 新增功能时，对应的配置项必须以 SQL 迁移语句形式给出，包含默认值。

## 三、代码层

1. 业务代码中禁止直接写死配置值，必须通过统一的配置读取函数获取：
   ```php
   config('配置键名', '默认值')
   ```

2. 配置读取函数必须自带缓存机制（Redis/内存缓存），避免每次请求查库。
3. 配置键名常量统一管理，禁止魔法字符串散落各处。

## 四、后台管理层

1. 每个新增配置项，必须同步在管理员后台的「系统配置」模块中提供可视化编辑界面。
2. 后台页面按 `config_group` 分组展示配置项，支持：
   - 布尔类型 → 开关组件
   - 数字类型 → 数字输入框（可带单位和范围校验）
   - 字符串类型 → 文本框/文本域
   - JSON 类型 → JSON 编辑器
3. 每个配置项下方显示 `remark` 说明文字，告知管理员该配置的作用和取值范围。
4. 保存配置后，必须自动清除对应缓存键，确保新配置实时生效。
5. 提供「恢复默认值」按钮。

## 五、缓存机制

1. 缓存键格式：`config:{config_key}`
2. 缓存过期时间：3600 秒
3. 后台修改配置 → 立即删除对应缓存 → 下次读取自动重建
4. 系统启动时可预加载全部配置到缓存

## 六、交付要求

你输出的完整功能代码，必须同时包含：

1. 业务逻辑代码（通过 `config()` 读取配置，无硬编码）
2. 配置项的 SQL 插入语句（带默认值和说明）
3. 后台配置页面的前端代码
4. 配置保存/清缓存的后端接口代码

## 七、验收标准

- 代码全文搜索：不得出现硬编码的魔法数字、魔法字符串（除了默认值参数）
- 修改后台配置后，业务逻辑立即生效，无需重启或重新部署
- 配置项有清晰的中文名称和使用说明，管理员无需懂代码也能操作

请基于以上规范完成开发，任何不符合规范的代码视为不合格。
